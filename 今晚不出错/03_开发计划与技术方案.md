# 03 开发计划与技术方案（48h Game Jam）

面向：全组。原则：**周六晚必须有人能从头玩到尾，哪怕丑。**

---

## 1. 技术选型

- **引擎**：纯 Web —— Vite + TypeScript + PixiJS（或直接 DOM+CSS，见下）。
  - 本作 90% 是文字排版 + 点击 + 拖拽，**推荐直接 DOM 实现**：提词器就是 HTML 文本流，词块是 `<span>`，卡拉OK高亮是 CSS 渐变裁切，拖拽用原生 DnD 或 pointer 事件。DOM 方案排版免费、调试快、文字清晰，比 Canvas 引擎更适合 48h。
  - 演播室画面区（主播动画、弹幕）用一个独立层：主播动画 = 3–5 张切换的立绘（正常/卡壳/喝水/咳嗽/红温），CSS 动画足够。
- **状态管理**：单一 GameState 对象 + 简单事件总线，不引框架 store。
- **数据**：全部 JSON 静态加载（见 02 文档），策划直接改 JSON 热重载。
- **音频**：Howler.js。news-babble（含混念稿声）用一段循环采样变速播放即可，不做 TTS（TTS 是 P2）。
- **部署**：静态托管（itch.io / GitHub Pages），零后端。

## 2. 模块拆分与接口

```
core/
├─ cursor.ts       # 朗读光标：逐词块推进、暂停/恢复、速度换算（字/秒→词块耗时）
├─ judge.ts        # 判定：点词命中、三选一结果、事故判定、台规命中注入
├─ rating.ts       # 收视率：事件→增减、红温阈值、死亡回调
├─ rules.ts        # 台规：announceAt 触发、hitTokens 注入 script、台规栏状态
├─ cards.ts        # 手牌：发放、倒计时、拖放判定、超时强插
├─ recorder.ts     # 事故记录器：所有事件打时间戳，供结算回放
ui/
├─ teleprompter.ts # 提词器渲染 + 词块点击 + 三选一浮窗
├─ studio.ts       # 主播立绘状态机 + 耳机气泡 + 弹幕层
├─ hud.ts          # 收视率条 / 台规栏 / 资源区 / 手牌区
└─ summary.ts      # 结算集锦页
```

- **接口契约（先定死，两人并行的前提）**：
  - `cursor` 每进入一个词块发事件 `onEnterToken(sentenceId, tokenIndex)`；`judge` 订阅并判事故。
  - `judge` 产出统一事件 `GameEvent {type, payload, ts}`，`rating`/`studio`/`recorder` 各自订阅，互不直接调用。
  - 一切表现（弹幕、立绘、音效）只订阅事件，不写在逻辑里。

## 3. 48 小时里程碑

### D1 上午（0–4h）：定案 + 骨架
- 全组过一遍 00/01/02 文档，锁定砍单（见 §4）。
- 程序：工程搭建，script.json 解析 + 提词器静态渲染（无光标）。
- 策划：开写一幕稿件（14 句 + 8 错），先交 3 句给程序当测试数据。
- 美术：定风格稿（演播室一屏 + 主播 5 态立绘规格）。

### D1 下午（4–10h）：核心循环闭环 ★最重要节点
- 光标推进 + 点词 + 三选一 + 事故判定 + 收视率增减跑通。
- **10h 检查点：能玩"一幕纯纠错"的灰盒版**。不到这个点，砍 P1 列表。
- 策划：一幕稿件完成，开写二幕 + rules.json。

### D1 晚（10–16h）：台规 + 卡片
- rules.json 注入机制、耳机气泡、台规栏。
- 赞助卡发放 / 拖放 / 超时。喝水、咳嗽。
- 弹幕层（先用文字池，不做美化）。
- 策划：二幕内容完，三幕冲突规则设计定稿。

### D2 上午（16–26h）：内容灌注 + 三幕
- 突发插播、红温表现、死亡流程。
- 三幕全部内容进游戏，**从头到尾完整一局**。
- 美术资产替换灰盒。

### D2 下午（26–36h）：结算 + 打磨
- 结算集锦页（recorder 回放 + 评分 + 称号 + 复制战绩）。
- 音效音乐、主播动画、弹幕爆发节奏。
- **34h 检查点：找 2 个没见过游戏的人盲测**，按 §5 调参。

### D2 晚（36–48h）：修 + 交
- 只修 bug 和数值，冻结功能。
- 部署、录宣传 GIF（选取"赞助词插进体育新闻"名场面）、写 itch 页面。
- 若有余力（>4h 空闲）才碰 P2。

## 4. 需求优先级（砍单顺序，从下往上砍）

| 级 | 内容 | 说明 |
|---|---|---|
| P0 | 提词器+光标+点词三选一+事故+收视率+一幕内容+简易结算 | 没有这些不叫游戏 |
| P0 | 弹幕层（哪怕纯文字飘过） | 可视化崩溃的最低配置，不可砍 |
| P1 | 台规系统（ban/duty/replace）+ 赞助卡 + 二三幕 | 游戏的灵魂，尽全力保 |
| P1 | 喝水/咳嗽、红温演出、事故集锦回放 | |
| P1.5 | 突发插播卡 | 若时间紧，三幕改为纯台规冲突 |
| P2 | 真人主播模式（音量检测） | 彩蛋，最后 4h 才考虑 |
| P2 | TTS 念稿、称号系统全量、简单模式开关 | 称号可先只做 3 个 |

## 5. 盲测调参清单（34h 检查点用）

按 01 文档设计红线逐条验证：

1. 测试者第一幕是否**不看说明**就学会点词三选一？（不行 → 首句强制引导：第一个错误词块闪烁）
2. 三幕的抱怨是"脑子跟不上"还是"手跟不上"？（后者 → 降光标速度 0.5，不减台规）
3. 有没有至少一次**笑出声**？记录在哪（通常应在：赞助词插进句中 / 天气预报白云被替换 / 弹幕嘲讽）。没有 → 加大 sponsor_fail 与荒诞命中的密度。
4. 死亡的玩家是否愿意立刻重开？（不愿 → 检查一幕是否太长、结算是否够爽）
5. 单局时长是否落在 8–12 分钟。

## 6. 团队分工建议（按 4 人组）

- **程序 ×2**：A 负责 core/（光标/判定/规则/卡），B 负责 ui/（提词器/演播室/结算）。接口契约（§2）D1 上午定死。
- **策划 ×1**：全部 JSON 内容 + 埋错写作（按 02 §7 规范）+ 数值调参。**内容量是本作最大风险，策划从第 1 小时就开始填表。**
- **美术 ×1**：演播室一屏、主播 5 态、UI 皮肤、结算页。风格建议：略带年代感的电视台质感（新闻联播蓝金 + 一点像素或粗描边喜剧化），主播表情是最重要资产。

## 7. 风险与预案

| 风险 | 概率 | 预案 |
|---|---|---|
| 内容量不足（埋错写不完） | 高 | 二幕句数 18→12；错误可复用题库改写；弹幕池先抄常见梗后润色 |
| 事实错误的"正确答案"本身出错 | 中 | 02 §7 第 6 条：双人交叉核实；盲测时专项请人挑刺 |
| 拖拽在触屏上手感差 | 中 | jam 版明示"桌面体验最佳"；卡片改为"点卡→点间隙"两连点作为降级方案 |
| 光标速度中文语速感不对 | 中 | 速度做成全局配置实时可调，盲测现场调 |
| 弹幕梗有版权/人身风险 | 低 | 语料池只用通用梗，不点名真实人物/媒体 |
