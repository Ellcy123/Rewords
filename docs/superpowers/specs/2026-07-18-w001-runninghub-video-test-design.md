# W001 RunningHub 图生视频试片设计

> 日期：2026-07-18
>
> 状态：方案已确认，等待书面规格审阅
>
> 目标：用一张自生成关键帧和一次 RunningHub LTX2.3 图生视频任务，验证《刷到你了》正式视频素材的制作链路

## 1. 试片范围

本次只制作第一关初始节点 `W001《婚礼灯架事故》` 的一条 9:16 竖屏试片，不修改剧情树、玩法配置或网页原型逻辑。

试片采用“关键帧图片 → RunningHub 图生视频 → 本地字幕与声音包装”的单镜头方案。首次付费任务只提交一次；仅当结果出现严重人物变形、画面基本静止或完全偏离婚礼事故时，才允许再提交一次。总计最多两次视频生成。

## 2. 叙事目标

试片必须符合项目既有短视频规则：第一秒出现事故，五秒内讲清“新娘因灯架坠落死亡，婚礼终止”，总时长不超过 15 秒，静音观看仍能理解主要结果。

原节点脚本为 12 秒，但本次生成主体控制在约 8 秒，以降低首测成本。生成视频负责事故动作和现场氛围；字幕、黑场、回放提示与线索在本地包装阶段完成。

## 3. 关键帧设计

关键帧由图像生成模型制作，输出为 9:16 竖屏 PNG。画面要求：

- 中国现代婚礼现场，精致拟真人短剧质感；
- 新娘位于画面视觉中心，身穿白色婚纱，刚刚抬头察觉危险；
- 四米高舞台灯架在她上方明显失稳并开始倾斜；
- 前景或两侧有惊慌宾客，但不能遮挡新娘与灯架关系；
- 用倾斜结构、坠落碎屑和人物反应表达事故，不出现血液或伤口；
- 构图为后续镜头后退、灯架继续坠落和现场混乱预留运动空间；
- 图片内不生成标题、字幕、平台 UI、水印或品牌标识。

生成后先人工检查手部、脸部、婚纱、灯架空间关系和可动画空间。只有关键帧通过检查后才上传 RunningHub。

## 4. 视频生成设计

使用已验证可访问的 RunningHub AI 应用：

- 应用：`【LTX2.3-图/文双模式视频】全自动台词设计`；
- `webappId`：`2031277395491164161`；
- 图片节点：`nodeId=3`、`fieldName=image`；
- 提示词节点：`nodeId=65`、`fieldName=text`；
- 时长节点：`nodeId=83`、`fieldName=value`；
- 模式节点：`nodeId=219`、`fieldName=value=false`，表示图生视频。

视频提示词描述以下连续动作：灯架从已失稳状态继续坠落；新娘抬头后本能后退；宾客惊呼并散开；镜头快速、稳定地向后撤；现场灯光闪烁，画面以强烈白闪收尾。动作应真实但不过度剧烈，不展示撞击人体、伤口或血腥内容。

首轮目标时长为 8 秒。若该应用仅接受特定时长，则使用最接近且不超过 10 秒的可用值，不为凑满原脚本而延长空镜。

## 5. 本地包装

RunningHub 原始结果下载后保留不动，另行输出游戏包装版。包装内容包括：

1. 开头 0～1 秒保留灯架坠落的最强动态，并加入撞击前的短促危险音；
2. 事故临界点使用闪白或短黑场，不展示直接伤害；
3. 添加“婚礼开始第 7 秒，新娘死亡”“婚礼未完成”等短字幕；
4. 结尾通过局部放大或短倒放强调高处卡扣，加入“这么高谁够得到？”评论线索；
5. 输出适合网页播放的 MP4，必要时另输出 WebM；
6. 音频响度保持克制，静音时字幕仍能独立传达剧情。

本次不要求生成完整配音。如果 RunningHub 返回可用环境音则保留并混音；若声音不可用，则本地使用撞击、宾客惊呼和现场环境声完成最低限度包装。

## 6. 文件组织

所有试片文件放入：

```text
刷到你了/assets/video-tests/W001/
├── W001_keyframe_v1.png
├── W001_ltx_raw_v1.mp4
├── W001_game_cut_v1.mp4
├── W001_thumbnail_v1.jpg
└── generation-notes.md
```

`generation-notes.md` 记录图片提示词、视频提示词、RunningHub 任务 ID、实际时长、生成次数、失败原因和最终采用版本。API Key 不写入该文件，也不进入 Git；调用时只从本地环境变量读取。

## 7. 数据流与失败处理

```text
读取 W001 脚本
→ 生成并检查关键帧
→ 上传图片到 RunningHub
→ 提交 AI App 图生视频任务
→ 轮询 taskId
→ 下载并验证原始视频
→ 本地包装字幕与声音
→ 输出网页可用文件和记录
```

失败处理规则：

- 上传失败：不提交任务，保留关键帧并重试上传；
- API 鉴权失败：停止，不把密钥打印到日志；
- 任务失败：记录 RunningHub 返回原因，再决定是否使用第二次额度；
- 严重变形或偏题：先收紧提示词，仅允许一次重试；
- 轻微瑕疵：优先通过裁切、闪白、字幕和剪辑处理，不额外烧生成次数；
- 下载或包装失败：保留任务 ID 和远端结果 URL，恢复后继续，不重新生成。

## 8. 验收标准

试片通过需同时满足：

- 9:16 竖屏播放无黑边；
- 第一秒能看出灯架事故正在发生；
- 五秒内能理解新娘死亡、婚礼终止；
- 新娘、婚纱、灯架和主要宾客没有影响理解的持续变形；
- 画面存在真实主体动作，不是单纯缩放静态图；
- 不出现血腥内容、乱码字幕、平台水印或第三方品牌；
- 静音观看包装版仍能理解核心事件和高处线索；
- 最终 MP4 能被常见桌面与手机浏览器播放；
- API Key 未写入素材、日志或 Git 提交。

## 9. 本次明确不做

- 不一次制作 W001 的全部 12 秒多镜头正式版；
- 不生成第二个剧情节点；
- 不把 RunningHub 调用接入玩家端或前端运行时；
- 不自动批量重试；
- 不修改关卡节点时长与现有剧情文档；
- 不为首次试片搭建通用视频生产后台。
