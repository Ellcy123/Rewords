# 《刷到你了》网页试玩原型设计规格

> 日期：2026-07-11  
> 状态：已完成设计讨论，等待规格审阅  
> 目标：制作一个可直接发给他人试玩、视觉接近成品的移动端优先网页演示版

## 1. 产品目标

《刷到你了》首个网页原型用于验证以下核心体验：

1. 玩家能否自然理解“从一条视频取得物品，再把它送进另一条视频”；
2. 跨频道物品是否能产生“事前想不到、事后说得通”的语义惊喜；
3. 正确与错误投放是否都能提供值得观看的短视频反馈；
4. 推荐流能否同时承担主界面、关卡地图和进度反馈；
5. 没有阅读设计文档的玩家能否在 10～15 分钟内完成第一关。

首版以手机浏览器为主要体验，同时兼容桌面浏览器。它是完整的对外试玩演示，而不是纯功能灰盒。

## 2. 设计原则

### 2.1 短视频平台优先

主界面高度还原用户熟悉的竖屏短视频交互语言，但不使用抖音名称、Logo 或专属品牌素材。玩家初看应认为自己进入了一个短视频 App，游戏系统只在购买、投放和时间线改写时显现。

### 2.2 内容频道彼此独立

三个初始频道使用不同的画面质感：

- 婚礼线：精致拟真人短剧，强调情绪可信；
- 古装线：高饱和、夸张、廉价竖屏短剧感；
- 知识线：低成本自媒体、手持拍摄和测评字幕感。

三类内容通过统一的平台 UI、商品系统和改命反馈连接，不能被包装成同一部动画的三个场景。

### 2.3 内容与玩法配置化

节点、物品和触发关系必须独立于 React 组件。增加或替换剧情节点时，应主要修改配置和素材，而不是改写推荐流或弹层代码。

### 2.4 原型范围可控

首版保留完整通关链，但不一次性制作全部 56 条分支。所有未进入原型的组合不显示在节点的可选礼物中，不提供“无效礼物”或“内容尚未制作”的破坏沉浸反馈。

## 3. 首版范围

### 3.1 内容节点

首版共包含 16 条内容节点：10 条正确骨架和 6 条精选错误结果。

正确骨架：

- 婚礼线：W001、W101、W200、W300、W301、W400；
- 古装线：C001、C101；
- 知识线：K001、K101。

错误结果：

| 触发 | 结果节点 | 目的 |
|---|---|---|
| W001 + 空调师傅 | X001《师傅到了但没有梯子》 | 展示条件缺失型失败 |
| W101 + 梯子 | X004《两把梯子还是不会修》 | 强化“工具不等于专业” |
| W300 + 空调师傅 | X012《空调师傅改修感情》 | 展示另一种人生 |
| W301 + 录音笔 | X016《证据翻录成了电音》 | 展示重复工具导致跑偏 |
| C001 + 梯子 | X021《王妃成了宫墙跑酷教练》 | 展示古装跨类型荒诞结果 |
| K001 + 空调师傅 | X028《两位师傅把电脑冻成冰箱》 | 展示知识区错误实验升级 |

每个可投放节点只显示已配置结果的 2～3 件“精选礼物”。这一限制是原型内容边界，不改变正式版四件物品完整组合矩阵的规划。

### 3.2 物品

首版使用四件关键物品：

1. 梯子；
2. 空调师傅；
3. 录音笔；
4. 投影服务。

物品均可重复购买。错误试验不能永久消耗唯一资源并造成软锁。

### 3.3 完整通关链

```text
C001 购买梯子
├─ 梯子 → W001 → W101
│  └─ 空调师傅 → W200 → 完播生成 W300
│
└─ 梯子 → K001 → K101
   └─ 购买录音笔
      ├─ 录音笔 → W300 → W301
      └─ 录音笔 → C001 → C101
         └─ 购买投影服务
            └─ 投影服务 → W301 → W400 → 通关
```

### 3.4 明确不做

首版不包含：

- 账号登录和云存档；
- 真实支付、广告和体力；
- 服务端 API；
- AI 实时生成视频或剧情；
- 真实社交分享；
- 完整 56 条分支；
- 第二关及长期账号经营；
- 复杂推荐算法；
- PWA 安装与离线缓存。

## 4. 玩家体验流程

```text
进入推荐流
→ 自由刷三条初始视频
→ 点击挂链查看商品
→ 购买物品
→ 在可互动视频打开“改命礼物”
→ 选择物品并确认投放
→ 播放 1～2 秒投放特效
→ 立即播放预制结果节点
→ 更新推荐流、背包和命运记录
→ 继续寻找跨视频关联
→ 播放 W400 并完成婚礼
```

首次结果必须立即播放，不能静默插入推荐流。正确结果结束后提示“时间线已改写”；错误结果结束后提示“发现另一种命运”，并返回仍可继续尝试的原问题节点。

## 5. 界面设计

### 5.1 推荐流

推荐流采用 9:16 全屏画布：

- 顶部：“关注 / 推荐”；
- 右侧：头像、点赞、评论、收藏、分享；
- 左下：账号、标题、字幕和商品卡；
- 底部：“推荐、命运记录、改命礼物、背包、我的”。

“改命礼物”占据固定位置。当前节点可接收物品时高亮，不可互动时保持原位并降低强调，避免布局跳动。

“我的”不包含账号系统，只提供试玩进度、声音设置、内容说明和重新开始入口。

### 5.2 底部抽屉

以下功能统一使用移动端常见的底部抽屉：

- 商品详情与购买；
- 评论列表；
- 改命礼物选择与投放确认；
- 命运记录；
- 背包与商品来源；
- 设置与重新开始确认。

任意时刻只允许一个抽屉打开。打开抽屉时暂停当前内容，关闭后从原位置继续。

### 5.3 教学

教学只解释操作，不公开谜题答案：

1. 首个商品卡轻微呼吸；
2. 首次购买后提示“物品可以送入其他视频”；
3. 首次刷到可互动节点时高亮“改命礼物”。

玩家完成相应操作后，该提示永久结束，不在每次重访时重复出现。

### 5.4 手机与桌面

手机端使用完整视口并适配安全区。桌面端居中显示最大约 430 × 932 的手机画布，外部使用低干扰深色背景。

输入方式：

- 手机：上下滑动换视频，单击暂停或继续；
- 桌面：滚轮、拖动、上下方向键换视频，单击暂停或继续；
- 打开弹层或投放结算时锁定换视频手势。

横屏时显示旋转设备提示，但不阻止继续操作。

## 6. 视频与声音

### 6.1 媒体抽象

每个节点允许使用以下任一种素材：

1. MP4/WebM 短视频；
2. 多张分镜图组成的动态短片；
3. 素材加载失败时的字幕动画备用片。

播放器只依赖统一的节点媒体接口，不能把剧情分支与某一种素材格式绑定。

### 6.2 首版画面方案

首版以“高质量分镜图 + 局部动画 + 字幕 + 音效”为主。局部动画包括镜头推拉、轻微景深、人物层位移、字幕打点、界面贴纸和结果特效。后续替换为正式短视频时不改变节点或触发数据。

所有内容遵循：

- 第 1 秒给出核心结果；
- 前 5 秒讲清事件；
- 总时长不超过 15 秒；
- 静音时依靠画面和字幕仍可理解。

### 6.3 声音层级

声音分为：

- 视频本体对白、环境和配乐；
- 滑动、购买、按钮和弹层 UI 音效；
- 礼物投放、时间倒转和节点解锁音效；
- 正确改命、错误命运和最终通关结算音效。

首次用户手势后才启用声音，以符合手机浏览器自动播放限制。界面常驻静音入口，页面进入后台时自动暂停媒体。

## 7. 推荐流规则

首版使用可控权重而非个性化算法：

1. 新生成节点首次强制播放；
2. 播放结束后加入推荐流前部；
3. 当前可推进节点获得较高权重；
4. 三个频道尽量交替出现；
5. 已完整观看的普通节点降低权重；
6. 错误结果只进入命运记录，不重复参与主推荐流；
7. 当前刚刚离开的节点不会立即再次出现。

排序必须可复现，避免测试和试玩过程中因为随机顺序无法定位问题。

## 8. 技术架构

### 8.1 技术栈

- React；
- Vite；
- TypeScript；
- CSS 负责布局、主题和主要动效；
- `localStorage` 负责单机存档；
- 静态构建，不依赖后端运行。

不引入重量级游戏引擎。第三方依赖仅在能显著减少手势、测试或兼容成本时采用。

### 8.2 目录边界

```text
刷到你了/web-prototype/
├─ src/
│  ├─ components/     通用 UI 与底部抽屉
│  ├─ features/       推荐流、商城、背包、改命、命运记录
│  ├─ content/        节点、物品、触发关系与评论配置
│  ├─ engine/         投放结算、解锁、排序与存档迁移
│  ├─ media/          分镜图片、视频、音频与封面
│  ├─ styles/         主题、动效、响应式与安全区
│  └─ tests/          单元和集成测试
└─ public/            直接加载的静态素材
```

每个模块只承担一个明确职责。UI 组件不直接修改游戏进度；所有状态变化通过引擎动作完成。

### 8.3 核心数据

`VideoNode` 至少包含：

- ID、频道、账号、标题和内容类型；
- 时长、封面和媒体描述；
- 字幕与评论；
- 商品链接；
- 可选礼物集合；
- 播放完成事件；
- 推荐权重参数。

`Item` 至少包含：

- ID、名称、价格和商品说明；
- 来源节点；
- 图标与视觉资源；
- 是否可重复购买。

`Trigger` 至少包含：

- 目标节点 ID；
- 物品 ID；
- 结果节点 ID；
- 正确、错误或资源推进类型；
- 状态变化；
- 是否消耗物品；
- 重复触发策略。

### 8.4 运行状态

存档只保存玩家行为与派生进度：

- 已观看和已解锁节点；
- 当前推荐流；
- 已发现商品；
- 背包数量与金币；
- 已触发组合；
- 当前婚礼阶段；
- 命运记录；
- 教学进度；
- 设置与静音状态；
- 存档版本号。

静态内容不复制进存档，避免剧情配置更新后存档体积和迁移复杂度增长。

### 8.5 结算流程

```text
接收投放请求
→ 验证当前没有其他结算
→ 验证节点、物品和库存
→ 查询 Trigger
→ 原子更新库存、节点、主线或命运记录
→ 持久化新状态
→ 播放投放特效和结果节点
→ 解除交互锁
```

状态更新必须是原子的。媒体播放失败不能回滚已经完成的触发，而应切换到备用分镜继续展示结果。

## 9. 经济与存档

### 9.1 试玩经济

- 开局金币足够完成首次教学购买；
- 首次观看新节点奖励少量金币；
- 首次发现错误命运同样奖励金币；
- 已发现商品可重复购买；
- 金币不足时提供“领取试玩金币”；
- 不设置真实货币、付费点或广告。

经济系统只用于形成“发现—购买—投放”的完整手感，不能阻断试玩主线。

### 9.2 存档

存档写入 `localStorage` 并带版本号。入口提供：

- 继续游戏；
- 重新开始；
- 清除存档；
- 通关后继续收集命运。

重新开始和清除存档必须二次确认。

## 10. 异常与防卡死

- 重复触发同一组合不扣物品、不生成重复节点；
- 素材失败时使用字幕动画备用片；
- 存档解析失败时把原始值备份到带时间戳的 `save-corrupt-*` 键，并提供安全重开；
- 存档版本无法迁移时明确提示重开，不允许白屏；
- 投放、弹层和换视频操作使用互斥锁，防止重复结算；
- 关键商品可重复购买；
- 试玩金币保证经济系统不造成软锁；
- 当前推荐流为空时从所有已解锁非终局节点恢复至少三条；
- 任何错误不得删除玩家已经解锁的商品来源。

## 11. 性能与兼容

目标浏览器：

- iOS Safari；
- Android Chrome；
- 桌面 Chrome、Safari 和 Edge 的当前主流版本。

性能策略：

- 仅挂载当前节点和前后各一个节点；
- 图片、音频和结果节点按需预加载；
- 同时只播放一个媒体源；
- 先展示封面和字幕，再等待高质量素材；
- 尊重系统“减少动态效果”设置；
- 触摸目标不小于 44 × 44 CSS 像素；
- 文本、字幕和按钮保持足够对比度。

## 12. 测试与验收

### 12.1 自动测试

单元测试覆盖：

- 正确、错误和资源推进 Trigger；
- 物品扣除与重复投放；
- 节点解锁顺序；
- 推荐流排序的可复现性；
- 存档序列化、恢复与版本迁移；
- 通关条件；
- 防软锁恢复。

浏览器集成测试覆盖：

- 首次购买与投放；
- 底部抽屉互斥；
- 手机滑动与桌面输入；
- 完整正确主线；
- 至少一个错误命运；
- 刷新后恢复进度；
- 重新开始。

### 12.2 人工验收

- 手机单手可完成全部操作；
- 玩家无需文档即可完成第一次购买和投放；
- 每条内容前 5 秒可理解核心事件；
- 滑动、弹层和投放无明显卡顿；
- 静音状态仍能理解剧情；
- 桌面端可使用滚轮、拖动和键盘完成通关；
- 刷新后进度正确；
- 首次完整通关时间控制在 10～15 分钟。

## 13. 最终交付

首版交付内容：

- 可运行的 React + Vite + TypeScript 项目；
- 手机优先、桌面兼容的完整试玩界面；
- 16 条内容节点与对应素材；
- 4 件物品及完整通关链；
- 商品、背包、投放、推荐流和命运记录；
- 本地存档和重置；
- 自动测试；
- 可部署的静态构建产物；
- 内容配置说明，支持继续补齐正式版分支。

## 14. 原型成功标准

原型被视为成功，必须同时满足：

1. 新玩家能独立完成首次购买和投放；
2. 新玩家能解释梯子在婚礼线和知识线中的两种含义；
3. 正确结果与错误结果均产生明确、立即、可理解的反馈；
4. 完整主线可以在 10～15 分钟内通关；
5. 手机与桌面均不存在阻断流程的交互问题；
6. 刷新、错误试验和金币消耗均不会造成软锁；
7. 内容节点能够通过配置继续扩展，而无需重写核心界面。
