# 《刷到你了》Demo 15 条真实视频批量制作设计

> 日期：2026-07-19
>
> 状态：方案已确认，等待书面规格审阅
>
> 目标：保留现有 W001、W101 真实视频，制作其余 13 条节点视频，使当前网页 Demo 的 15 个节点全部使用真实 MP4，并继续支持 CSS 分镜回退

## 1. 范围

当前 `web-prototype/src/content/nodes.ts` 实际登记 15 个节点。W001、W101 已有真实视频，本次制作并接入以下 13 条：

- 婚礼线：W300、W301、W400、X001、X004、X012、X016；
- 古装线：C001、C101、X021；
- 知识线：K001、K101、X028。

本次不扩展到完整 56 条策划树，不新增节点，不修改触发关系、物品经济、通关条件或推荐流规则。README 中“16 条内容节点”的旧描述应随集成改为实际的 15 条。

## 2. 制作方案

采用“三套视觉连续性”方案，而不是让 13 条视频彼此独立生成。

### 2.1 婚礼视觉包

沿用 W001、W101 已建立的中国现代婚礼现场、暖色灯光、白色婚纱新娘、黑色舞台灯架和拟真人移动短剧质感。新增婚礼节点尽量继承同一新娘身份、场地结构、服装配色和灯光方向。

W300、W301、W400 组成连续的获救、造谣、举证与婚礼完成段落。X001、X004、X012、X016 是同一世界的错误结果，可以改变动作和道具，但不改变主要角色和场地识别。

### 2.2 古装视觉包

建立统一的年轻王妃、冷宫宫墙、宫廷色彩和竖屏古装短剧质感。C001 与 X021 共享冷宫及宫墙空间；C101 转入金銮殿，但保留同一王妃身份和服装识别。

### 2.3 知识视觉包

建立统一的中国年轻知识博主、桌面电脑、空调实验空间和冷色科技工作室。K001、K101、X028 共享博主身份和工作室结构，通过结冰、VPN 界面隐喻和双空调实验区分剧情。

## 3. 单条视频结构

每条视频使用一张严格 9:16 的关键帧驱动约 8 秒图生视频。关键帧必须在付费调用前通过人物、手部、关键道具、空间关系和无文字检查。

视频主体遵守以下节奏：

1. 0～1 秒出现核心动作或异常；
2. 1～5 秒完成因果转折；
3. 5～8 秒呈现结果、笑点或线索；
4. 结尾采用稳定停点、白闪、遮挡或自然运动收束，避免展示血腥和直接人体撞击。

复杂节点不增加付费镜头数量，而是在一个连续镜头内用人物动作、镜头运动和网页字幕完成叙事。原始视频不烧录游戏字幕，不生成平台 UI、品牌、水印或不可控文字。

## 4. 生成工作流

继续使用已经验证的 RunningHub LTX2.3 图生视频 AI App：

- WebApp ID：`2031277395491164161`；
- 图片节点：`3/image`；
- 提示词节点：`65/text`；
- 时长节点：`83/value=8`；
- 模式节点：`219/value=false`。

每个节点的数据流为：

```text
读取节点脚本与现有视觉参考
→ 生成或编辑 9:16 关键帧
→ 付费调用前视觉检查
→ 上传关键帧
→ 提交一次 RunningHub 图生视频任务
→ 轮询并下载原始 MP4
→ ffprobe 与抽帧检查
→ 必要时使用唯一一次重试
→ 生成缩略图和字幕 JSON
→ 复制网页媒体并登记节点配置
```

API Key 只从当前进程环境或安全交互输入中读取，不写入脚本、素材、日志、设计、计划或 Git 历史。签名上传地址和下载地址也不进入生成记录。

## 5. 任务与重试上限

13 条新视频各默认提交一次付费任务。仅出现以下任一严重问题时，允许该节点重试一次：

- 主体人物或关键道具在主要动作段持续严重变形或消失；
- 核心动作没有发生，导致因果关系无法识别；
- 场景、角色或事件完全偏离节点脚本；
- 视频基本静止，仅有画面缩放；
- 结果无法完整解码或不符合浏览器媒体规格。

普通运动模糊、短暂手部瑕疵或不影响理解的背景漂移不触发重试。整个批次最多提交 26 次付费视频任务；达到上限后不再自动付费调用，而是优先通过裁切、字幕时序、缩略图或回退分镜处理。

## 6. 文件组织

每个新节点使用独立目录：

```text
刷到你了/assets/video-tests/<NODE_ID>/
├── <NODE_ID>_keyframe_v1.png
├── <NODE_ID>_ltx_raw_v1.mp4
├── <NODE_ID>_thumbnail_v1.jpg
├── <NODE_ID>_captions.json
└── generation-notes.md
```

如果发生重试，保留实际采用的版本，并在 `generation-notes.md` 中记录提交次数、采用版本和拒绝原因。记录包括关键帧提示词、视频提示词、非敏感任务 ID、消耗、运行时间、媒体参数和验收结论。

网页运行时副本统一放入：

```text
刷到你了/web-prototype/public/media/
```

源素材目录与网页副本使用稳定、可预测的文件名，复制后用 `cmp` 验证字节一致。

## 7. 字幕与前端集成

每条视频本体保持无游戏字幕。字幕时间轴以 JSON 保存，并转换或登记为节点的 `media.captions` 配置。播放器继续依据 `<video>.currentTime` 选择字幕，不建立独立计时器。

每个新节点配置：

- `/media/<NODE_ID>_ltx_raw_v1.mp4`；
- `/media/<NODE_ID>_thumbnail_v1.jpg`；
- 连续、不重叠且不超过真实视频时长的字幕时间轴。

现有媒体加载或播放失败时的 CSS `FallbackStoryStage` 保留。真实视频接入不得改变礼物交互、推荐流、记录页、暂停逻辑和通关触发。

## 8. 验收标准

每条新视频必须同时满足：

- 9:16 竖屏，无黑边，主要主体不被网页交互安全区遮挡；
- H.264 视频与 AAC 音频，可被常见桌面和手机浏览器播放；
- 时长约 8 秒，文件能完整解码；
- 前三秒能识别节点核心结果，五秒内能理解主要因果；
- 人物和关键道具没有影响理解的持续严重变形；
- 有真实主体动作，不是静态图缩放；
- 无血腥、直接人体撞击、乱码、品牌或水印；
- 静音时配合网页字幕能理解剧情；
- API Key 和签名 URL 未进入项目文件或 Git。

整批集成完成后运行：

- 所有 15 个节点均存在有效 `media` 配置的内容测试；
- 字幕区间、媒体路径和节点 ID 的内容校验；
- `ffprobe` 批量媒体规格与可解码检查；
- `npm test`；
- `npm run typecheck`；
- `npm run build`；
- 浏览器中抽查三条世界线的播放、暂停、循环、字幕和 CSS 回退。

## 9. 失败与恢复

- API Key 不可用或鉴权失败：停止付费阶段，保留已完成的提示词、关键帧和清单；
- 上传失败：重试上传，不重新生成关键帧；
- RunningHub 任务失败：记录非敏感错误，只消耗该节点允许的一次重试；
- 下载失败：保留任务 ID，恢复下载，不重新提交生成；
- 本地包装或前端集成失败：保留原始 MP4，从本地步骤继续；
- 单条视频最终不可用：该节点继续使用 CSS 分镜回退，不阻塞其他节点制作，但整批不得标记为全部完成。

## 10. 明确不做

- 不制作当前 Demo 之外的另外 41 条策划视频；
- 不重新生成已经验收的 W001、W101；
- 不搭建玩家端或线上运行时视频生成服务；
- 不自动突破 26 次付费任务上限；
- 不烧录游戏字幕，不引入配音制作范围；
- 不修改核心玩法、剧情树、触发表和物品系统。
